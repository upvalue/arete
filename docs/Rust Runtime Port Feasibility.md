# Rust Runtime Port Feasibility

This note studies whether Arete's current C++ runtime can be ported to Rust
while dropping the template JIT, native VM, SDL, libuv, and other non-core
features at first. The short answer is yes, but the first useful Rust port
should target semantic compatibility and bootstrapping, not C++ ABI or heap
image compatibility.

## Current Rust Prototype State

As of 2026-05-16, the port is no longer only a feasibility study. A parallel
Rust executable exists and the C++ implementation remains untouched as the
oracle/runtime baseline.

New Rust-side entry points:

- `rust/arete-rs/`: standalone Rust crate with no external dependencies yet
- `bin/arete-rs`: generated by `make bin/arete-rs`
- `make test-rust`: builds `bin/arete-rs` and runs the Rust snapshot suites
- `utils/run-rust-tests.py`: snapshot runner for Rust `reader` and `preboot`
  suites

Verified state:

- `make test-rust` passes 36/36:
  - `tests/reader`: 17/17
  - `tests/preboot`: 19/19
- `python3 utils/run-tests.py` passes 105/105 against the existing C++ binary.
- `cargo fmt`/`rustfmt` are not installed in this environment, so formatting is
  currently maintained manually.

Implemented Rust surface:

- Rust-native `Value` enum and object enum.
- Reader coverage for the existing reader snapshots: numbers, strings,
  characters, symbols, keywords, pairs, dotted lists, vectors, expression
  comments, nested block comments, parameter objects, and renaming quasiquote.
- Writer coverage for reader snapshots plus enough shared-structure printing
  for the current `pretty-print` preboot test.
- Direct evaluator for the preboot language subset: `quote`, `if`, `begin`,
  `define`, function-definition sugar, `set!`, `lambda`, `let`, function
  application, closures, mutable captures, and a tail-position loop that handles
  the current TCO smoke test.
- Core preboot builtins for numbers, lists, mutation, equality, printing,
  strings, tables, records, file/string ports, `read`, exceptions, `try`,
  `unwind-protect`, and `value-bits`.

Important limitations:

- The current Rust object store uses `Rc<RefCell<Object>>`. This is a semantic
  scaffold, not the planned Arete heap. It has no moving collector, no root
  protocol, and no allocation safepoints.
- There is no macro expander, module system, compiler, bytecode VM,
  `OpenFn->procedure`, image save/load, bytevector support, boxes, source
  locations, full CLI parity, or REPL.
- The Rust runtime does not yet load `boot.scm` or `scheme/*.scm`.
- Exception support is sufficient for `tests/preboot/try.scm` and
  `tests/preboot/unwind.scm`, but not yet a complete model of Arete
  continuation/error behavior.
- Tables are a deterministic vector-backed implementation for test parity, not
  a final hash table.

## Current Runtime Shape

The native runtime is small enough to port deliberately:

- `src/*.cpp`: about 19k lines, including the JIT and native VM
- `arete.hpp`: about 2.9k lines
- boot Scheme in `scheme/*.scm`, `boot.scm`, `compile.scm`: about 4.4k lines
- current tests: `python3 utils/run-tests.py` passes 105/105 in this checkout
- low-level doctest runtime/GC tests: `tests/test-semispace` passes 18/18
- temporary bootstrap image: `bin/arete boot.scm --save-image /tmp/arete-feasibility.heap.boot`
  succeeds and produced a 947kb image in this checkout

The runtime boundary is concentrated in:

- `arete.hpp`: tagged `Value`, heap object layouts, `State`, GC root protocol,
  and VM opcode numbers
- `src/gc.cpp`: moving semispace collector, root tracing, finalizers, perf
  counters
- `src/eval.cpp`: tree-walking interpreter used by early bootstrap and macro
  execution
- `src/builtins.cpp`, `src/numbers.cpp`, `src/files.cpp`, `src/table.cpp`,
  `src/strings.cpp`, `src/platform.cpp`: core host functions
- `src/reader.cpp`, `src/writer.cpp`: syntax input/output
- `src/vm.cpp`: bytecode VM, the runtime target of the self-hosted compiler
- `src/image.cpp`: heap image save/load

The current value representation is C-like and layout-sensitive:

- fixnums use low bit `1`
- constants use low bits `10`, with `#f` as zero
- heap pointers are aligned and use low bits `00`
- heap object headers contain type, flags, mark/forwarding state, and a small
  header integer
- the collector moves objects and updates every known slot

The compiler produces `VMFunction` objects through `OpenFn->procedure`, copying
Scheme-side instruction vectors into a `Bytevector` of `size_t` opcodes. There
are currently 56 VM opcodes. The native VM and template JIT are optional
accelerators; the bytecode VM in `src/vm.cpp` is the correctness target.

There are about 177 core `AR_DEFUN` host bindings when excluding the JIT,
native VM, SDL, and libuv groups, and 212 total with those optional groups.

## Feasible Initial Scope

The first Rust milestone should be:

1. Implement the same logical `Value` universe: constants, fixnums, flonums,
   strings, chars, bytevectors, symbols, renames, pairs, vectors, tables,
   records, exceptions, interpreted functions, C/Rust host functions,
   VMFunctions, closures, boxes, and ports.
2. Port `State` boot and builtins needed by `boot.scm` through compiler
   installation.
3. Port the reader, writer, tree-walking interpreter, and core host functions.
4. Port `OpenFn->procedure` and `apply_vm`.
5. Run `boot.scm`, save a Rust-native image, reload it, and pass the existing
   Scheme test runner.

This can drop:

- `src/compile-x64.cpp.dasc` and `src/compile-x64.cpp`
- `src/vm-native-x64.cpp.dasc` and generated native dispatch
- native VM builtins and stats
- SDL/libuv initially
- Emscripten/distilled build paths initially

The current `heap.boot` file format should not be a first milestone. It writes
raw C++ heap layouts with pointer-size and object-layout assumptions. A Rust
port can bootstrap from `boot.scm` and write a Rust-native image. Compatibility
with existing C++ images is possible only if the Rust port preserves exact
object layout and pointer update semantics, which would fight most ecosystem
runtime design choices.

## GC Decision

Use a hand-rolled precise stop-the-world semispace collector for the first
Rust runtime. Arete's current C++ collector is deliberately simple and has
already beaten a previous lower-pause collector attempt on this workload.
That points toward preserving the collector shape, not adopting a lower-pause
non-moving collector whose performance profile is wrong for allocation-heavy
Scheme boot/compiler code.

The first collector should optimize for:

- bump-pointer allocation on the hot path
- collection cost proportional to live data, not allocated garbage
- compacted object locality after collection
- precise roots and slots, no conservative stack scanning
- simple heap validation and stress-collection modes
- complete control over object representation and image serialization

This is not a line-for-line port of the C++ collector. The Rust runtime can
choose a new `Value` and object layout, but the GC algorithm should remain a
copying semispace until benchmarks prove another design wins.

MMTk remains a possible later integration point if Arete becomes a platform for
collector experimentation, but its binding work is larger than the initial
source-compatible runtime. Other Rust GC crates are written off for this phase:
they either optimize for lower pauses, stable addresses, conservative roots, or
general Rust object graphs rather than Arete's known allocation-throughput
profile.

## Heap Strategy

Use a small abstraction boundary from day one:

```text
Value -> HeapValue pointer/reference
State roots -> Heap::collect()
TraceObject -> visit outgoing Value slots
Store API -> write heap references through one place
```

The abstraction is not meant to make multiple GC backends equally supported on
day one. It is meant to keep allocation, tracing, roots, and pointer stores from
leaking through the entire interpreter and VM.

Design constraints:

- no raw object reference may survive an allocation unless it is rooted or
  recoverable from a stable VM stack slot
- every heap object must implement explicit tracing
- globals, temporary host-call values, the symbol table, module tables, and the
  VM stack are roots
- open closure boxes must either point to GC-traced VM stack slots or close over
  a normal heap value before the owning frame disappears
- heap image serialization should use logical object identities, not raw
  addresses

Current-state caveat: `arete-rs` does not yet have this boundary. The
`Rc<RefCell<Object>>` store was useful for proving reader/preboot semantics
quickly, but it should not leak into the expander/VM/GC design. Before the Rust
runtime depends on compiled code or long-running bootstrap state, introduce a
real heap module and move object allocation, tracing, roots, and pointer stores
behind it.

## Source-Level Compatibility Reorientation

If exact runtime concepts are a non-goal, this becomes a cleaner project:
write a source-compatible Arete Scheme in Rust, using the C++ runtime only as a
behavior oracle and benchmark baseline. Do not preserve `Value` bits, heap
object layout, bytecode encoding, `heap.boot`, `AR_FRAME`, native-code hooks,
or the current VM stack protocol unless they independently prove useful.

The compatibility contract should move up to:

- reader syntax and source-location behavior observable in errors
- printed output for supported data
- core special forms and macro/module behavior
- public builtins used by `boot.scm`, `scheme/*.scm`, examples, and tests
- `eq?`, `eqv?`, `equal?`, mutation, symbol interning, and object identity
  semantics at the Scheme level
- ability to load the existing Scheme bootstrap sources without translating
  them

This changes several design choices.

`Value` can be a Rust-native enum rather than a tagged word:

```text
Value =
  Bool | Nil | Undefined | Eof | Fixnum(i64) | Flonum(f64) | Char(char)
  | Symbol(SymbolId)
  | Object(HeapRef)

Object =
  Pair | Vector | String | Bytevector | Table | Record | Function
  | VMFunction | Closure | Box | Rename | Exception | Port | ...
```

That is slower than a carefully packed tagged word in the abstract, but it is
much easier to implement correctly, easier to fuzz, and still leaves room for a
later packed representation once the compatibility suite is green. Scheme-level
identity can come from interned symbol IDs plus heap allocation identity; it
does not require exposing raw addresses.

The heap can be moving from the start. Source compatibility does not require
stable object addresses, and Arete already has a working moving-collector
discipline. Object identity should be represented by object equality/reference
identity inside the runtime, not by exposing raw addresses to Scheme code.
Ports and other external resources should use handles that can survive object
movement.

The VM can change format. The compiler can still be the existing
`scheme/compiler.scm`, but `OpenFn->procedure` could translate its instruction
vector into a Rust enum bytecode or a compact Rust-owned instruction array.
Opcode numbers only need compatibility at the `OpenFn->procedure` boundary;
the in-memory VM representation can be whatever is easiest to execute and
debug.

The image format should definitely change. If the implementation is
source-compatible only, the right image is a Rust-native serialized snapshot or
no image at all until bootstrap works. Rebuilding from `boot.scm` is slower but
keeps the first milestone honest.

The development sequence should be contract-first:

1. Define a compatibility runner that compares Rust Arete against C++ Arete on
   `.scm` inputs: stdout, stderr class, exit status, and optionally normalized
   source locations. The current `utils/run-rust-tests.py` is a snapshot runner,
   not yet a full differential runner.
2. Port the reader/writer and pass `tests/reader` by snapshot comparison.
   Prototype status: done for the existing 17 reader snapshots.
3. Implement a minimal evaluator and enough builtins to pass `tests/preboot`.
   Prototype status: done for the existing 19 preboot snapshots.
4. Load existing `scheme/expand.scm` and pass expander tests.
5. Load existing compiler sources and translate `OpenFn->procedure` into the
   new Rust VM representation.
6. Only after that, choose whether the Rust VM needs tagged values, direct
   threading, specialization, or semispace tuning.

The major benefit is that large parts of the C++ runtime stop being design
constraints. The major risk is semantic drift: without exact runtime concepts,
the compatibility harness has to become much stronger. Every implementation
shortcut should be judged against the existing tests plus differential runs
against the C++ binary.

## Ecosystem To Use

The port should still lean on Rust libraries outside the GC core:

- `clap` for CLI parsing instead of hand-rolled argument handling
- `rustyline` for REPL line editing instead of vendored linenoise
- `hashbrown` or `indexmap` for tables and deterministic test-friendly maps
  where order matters
- `serde` with a deliberate binary format for Rust-native images
- `thiserror` or `miette` for structured runtime/read/expand error plumbing
- `insta` for reader/writer/compiler snapshot tests
- `proptest` for reader and optimizer differential tests
- `criterion` for microbenchmarks equivalent to `bench/interp/`

Do not outsource the Scheme reader/parser until the existing syntax behavior is
well documented. Arete's reader carries source-location and syntax quirks that
are more important than parser-combinator reuse.

The current prototype intentionally has zero dependencies. That is acceptable
for the first semantic checkpoint. Add ecosystem crates when their ownership
area becomes active: `clap` for real CLI parity, `indexmap` or `hashbrown` for
tables, `serde` for images, and `criterion`/`proptest` once the relevant
surfaces exist.

## Main Risks

Rooting and safepoints are the largest correctness risk. The C++ runtime uses
`AR_FRAME`, handles, globals, temps, the symbol table, native frames, protected
argument arrays, and the VM stack. A Rust port needs one root story that covers
all of these without relying on accidental Rust stack liveness.

The custom collector is a maintenance risk. Arete will own heap growth policy,
copying/tracing bugs, stress modes, finalization behavior, and future
generational work. This is accepted because the first target collector is small,
because Arete needs control over representation, and because the existing
workload evidence favors the existing semispace shape.

The image format is a portability risk. Reusing C++ images ties Rust to C++
layout. A Rust-native image format avoids that and is the pragmatic path.

The VM stack and mutable-capture boxes are a semantic risk. Boxes can refer to
open VM stack slots and later close over a copied value. This behavior must be
ported with tests before trusting closure-heavy compiler output.

Host builtins are a volume risk. The core set is not conceptually hard, but it
is broad enough that mechanical porting and snapshot coverage matter.

Performance expectations should be staged. Existing tickets show that some
real workloads are VM-call dominated, not GC dominated. Rust will not
automatically fix that. A clean Rust bytecode VM may be easier to optimize, but
the same apply/frame/arity/closure costs must be measured.

## Milestones

Completed prototype milestones:

1. Rust crate skeleton with `Value`, object enum, symbols, pairs, vectors,
   strings, and a printer.
2. Reader and writer snapshot parity for `tests/reader`.
3. Minimal global environment and builtin registry.
4. Tree-walking interpreter enough to pass `tests/preboot`.

Remaining milestones:

1. Replace the snapshot-only Rust runner with a differential runner that can
   compare Rust and C++ on selected suites and normalize expected error output.
2. Fill the missing source-language runtime surface needed by
   `scheme/expand.scm`, `scheme/syntax.scm`, and `scheme/types.scm`.
3. Load the existing expander sources and pass `tests/expander`.
4. Fill module/library support enough to load `scheme/library.scm` and module
   tests.
5. Port `OpenFn->procedure` and `apply_vm`; compile and run compiler tests.
6. Introduce the real heap/root abstraction and then the precise semispace
   collector. The current `Rc<RefCell<_>>` scaffold should disappear before
   bootstrap is considered mature.
7. Bootstrap full `boot.scm`; save and reload a Rust-native image.
8. Run a Rust equivalent of `python3 utils/run-tests.py` and add low-level
   heap/collector tests comparable to `tests/test-semispace`.
9. Benchmark against C++ on `boot`, `bootstrap-and-psyntax`, `fib`, `tak`,
   `nboyer`, `peval`, and `gcbench`.

## Multi-Agent Handoff Plan

All agents should treat the C++ runtime as a read-only oracle unless a task
explicitly says otherwise. Keep write scopes narrow. Every completed handoff
should leave `make test-rust` green, and any task touching existing Scheme/C++
behavior should also run the comparable C++ suite.

### Wave 1: Parallel Now

Agent A: compatibility harness.

- Write scope: `utils/run-rust-tests.py`, new `utils/*rust*` harnesses,
  `GNUmakefile`.
- Goal: add a differential mode that runs `bin/arete` and `bin/arete-rs` on the
  same files, compares stdout, stderr class, and exit status, and supports
  suite selection.
- Acceptance: existing `make test-rust` still passes; new differential runner
  passes on `tests/reader` and `tests/preboot`.

Agent B: reader/writer completeness and diagnostics.

- Write scope: `rust/arete-rs/src/reader.rs`,
  `rust/arete-rs/src/printer.rs`, reader-focused tests.
- Goal: move beyond positive snapshots by adding error-output normalization,
  source-location storage, shared structure syntax if needed, bytevector
  syntax, and writer behavior required by expander/compiler debugging.
- Acceptance: current reader suite stays green, negative reader cases are
  checked intentionally, and Rust/C++ reader output remains compatible for
  supported syntax.

Agent C: builtin inventory for expander boot.

- Write scope: documentation or a generated report first; Rust runtime changes
  only after inventory is clear.
- Goal: run/inspect `boot.scm`, `scheme/expand.scm`, `scheme/syntax.scm`, and
  `scheme/types.scm` to list the exact missing Rust special forms, predicates,
  and builtins needed before the expander can load.
- Acceptance: a checklist grouped by runtime area with file/function references
  and a recommended implementation order.

Agent D: heap abstraction spike.

- Write scope: new Rust modules such as `heap.rs`/`trace.rs` plus focused Rust
  tests. Avoid broad rewrites of `runtime.rs` until the API is reviewed.
- Goal: design the allocation, root, trace, and store APIs that can replace the
  current `Rc<RefCell<Object>>` scaffold.
- Acceptance: standalone heap tests for allocation, object identity, tracing
  shape, and stress collection API stubs; no regression in `make test-rust`.

### Wave 2: After Inventory

Agent E: expander-runtime builtins.

- Write scope: `rust/arete-rs/src/runtime.rs`,
  `rust/arete-rs/src/value.rs`, and small supporting modules.
- Dependencies: Agent C's inventory.
- Goal: implement the missing host functions and predicates needed to load
  `scheme/expand.scm`, `scheme/syntax.scm`, and `scheme/types.scm`.
- Acceptance: Rust can load those files far enough to run a first selected
  `tests/expander` smoke test, with current reader/preboot tests still green.

Agent F: expander execution.

- Write scope: Rust runtime and expander-specific compatibility tests.
- Dependencies: Agent E.
- Goal: make the self-hosted expander execute under Rust and pass
  `tests/expander`.
- Acceptance: Rust expander suite parity for all currently enabled expander
  snapshots.

Agent G: ports, files, CLI, and module path parity.

- Write scope: `rust/arete-rs/src/main.rs`, runtime port/file support, CLI
  tests, and possibly `Cargo.toml` for `clap`/`rustyline`.
- Goal: support the command-line and port behavior required by boot/module
  loading without copying C++ image compatibility.
- Acceptance: Rust can load multiple files in the same process with CLI options
  needed by the test runner; file/string ports match supported C++ behavior.

### Wave 3: After Expander

Agent H: compiler and VM boundary.

- Write scope: new Rust VM/compiler-boundary modules plus runtime integration.
- Dependencies: expander execution.
- Goal: implement `OpenFn->procedure`, translate compiler instruction vectors
  into Rust-owned bytecode, and implement `apply_vm`.
- Acceptance: selected `tests/compiler` files pass under Rust, then broaden to
  the full compiler suite.

Agent I: real collector integration.

- Write scope: heap modules and runtime allocation integration.
- Dependencies: Agent D API, enough runtime stability to test meaningful
  programs.
- Goal: replace `Rc<RefCell<Object>>` with the planned precise semispace heap,
  including root registration for globals, temporary host values, ports, and VM
  stack once it exists.
- Acceptance: `make test-rust` passes with a stress mode that collects at every
  allocation; later, expander/compiler suites pass under stress collection.

Agent J: Rust-native image format.

- Write scope: new image serialization modules and CLI hooks.
- Dependencies: stable object model and collector rooting.
- Goal: serialize logical object identities, not raw addresses, and reload a
  Rust-native heap image.
- Acceptance: bootstrap state can be saved, reloaded, and used to run the same
  post-load smoke tests.

## Recommendation

A Rust port is feasible and should be easier to maintain than the current C++
runtime once it is through the bootstrap barrier. The feasible path is not a
line-for-line rewrite with a third-party GC swapped underneath. It is a staged
runtime port with a custom precise semispace collector, an explicit root
discipline, a correctness-first bytecode VM, and a Rust-native image format.

Hand-roll the GC first. The collector is small enough, the performance profile
matches Arete's existing workload, and owning it preserves freedom over `Value`,
object layout, roots, and image serialization. Drop the template JIT, native VM,
SDL, libuv, and C++ heap-image compatibility until the Rust runtime can
bootstrap and pass the existing test matrix.
