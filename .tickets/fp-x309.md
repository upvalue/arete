# Stabilize psyntax bootstrap stress test

Psyntax is useful in this repo as a high-stress bootstrap workload: it loads a large preprocessed syntax-case system into a booted Arete image and asks it to expand its own source. This path exposed multiple runtime issues, including invalid current-module restoration after load, stale assumptions in the experimental bootstrap script, and VM recursion through tail-position builtin apply.

## Design

Keep docs/Psyntax.md focused on what psyntax is and why it is a good stress test. Track transient debugging and fix history in this ticket instead of the wiki. Keep the working bootstrap command small and aligned with boot.scm rather than manual reloading of compiler pieces.

## Acceptance Criteria

bootstrap-and-psyntax.scm completes. Full-file sc-expand of tests/psyntax/psyntax.ss completes. docs/Psyntax.md describes psyntax as a stress test / benchmark and links to this ticket rather than embedding ephemeral incident history.


## Notes

**2026-04-12T17:08:50Z**

Failure sequence and implemented fixes:

- Initial failure was before psyntax itself loaded: bootstrap-and-psyntax.scm reset *current-module* into a bad state after boot.scm.
- Root cause: src/files.cpp restored G_CURRENT_MODULE unconditionally after load; when the saved value was not a module table, bootstrapping left the process in an invalid module state.
- bootstrap-and-psyntax.scm also contained stale manual reload steps for scheme/syntax.scm, scheme/compiler.scm, and pull-up-bootstraps; simplifying it to boot.scm + psyntax load path matched the current system better.
- After bootstrap issues were cleared, psyntax exposed a VM/runtime bug: tail-position builtin apply recursively re-entered the VM through fn_apply -> State::apply instead of preserving tail calls.
- Minimal reproducer: (let loop ((n 10000)) (if (= n 0) 'ok (apply loop (list (- n 1)))))
- src/vm.cpp now flattens builtin apply in OP_APPLY_TAIL and turns it into a real VM tail call when the underlying target is a Scheme procedure.
- Additional VM hardening kept from the investigation: zero-arg tail-call handling via state.temps, safer upvalue close/get/set behavior on the code paths exercised by psyntax.
- scheme/expand.scm was also tightened so already-qualified symbols stay qualified and env-lookup destructuring is explicit on key lookup paths.

Verification:
- ./bin/arete bootstrap-and-psyntax.scm completes.
- ./bin/arete boot.scm --eval "(set-top-level-value! 'COMPILER-WARN-UNDEFINED #f)" --eval "(load \"tests/psyntax/load.scm\")" --eval "(begin (for-each1 sc-expand (slurp-file \"tests/psyntax/psyntax.ss\")) (print 'all-ok))" prints all-ok.
- ./bin/arete boot.scm --eval "(let loop ((n 10000)) (if (= n 0) 'ok (apply loop (list (- n 1)))))" completes without native stack overflow.

**2026-04-12T17:14:46Z**

bootstrap-and-psyntax.scm also had a shape-dependent direct-entry crash: the same body succeeded via --eval but segfaulted as a multi-form file. Wrapping the post-boot.scm work in a single begin fixed the direct ./bin/arete bootstrap-and-psyntax.scm path.
